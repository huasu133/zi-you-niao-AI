# 主流商业 AI 产品记忆系统调研报告

> 调研日期：2026-06-03
> 调研范围：ChatGPT、Claude（Projects + Code）、WorkBuddy、Cursor / GitHub Copilot、Replit Agent

---

## 一、ChatGPT Memory

### 1.1 架构概览

ChatGPT 记忆系统是一个**产品层外挂系统**，而非模型本体能力。采用**四层上下文架构 + 六模块存储**设计。核心思路：为每个用户构建**人格镜像（Persona）**，在每次请求时从画像中提取相关信息注入 System Prompt。

### 1.2 四层上下文架构

| 层级 | 存储方式 | 注入时机 | 淘汰策略 |
|------|----------|----------|----------|
| **会话元数据** | 临时存储，不持久化 | 会话开始时一次性注入 | 会话结束后全部清除 |
| **用户记忆**（显式） | 长期持久化存储 | 每条消息都注入，所有已存储事实完整放入 | 仅由用户主动删除或模型移除 |
| **最近对话摘要** | 轻量级摘要（~15条），仅含用户消息 | 推断为每次会话注入 | 滑动窗口淘汰旧摘要 |
| **当前会话消息** | 完整保留全部消息原文 | 每条消息包含滑动窗口历史 | Token 数量上限触发淘汰 |

### 1.3 六模块 JSON 存储结构

逆向工程揭示的六模块 JSON 档案：

| 模块 | 内容 | 用户可见/可编辑 |
|------|------|:---:|
| `Model Set Context` | 用户明确要求记住的事实（带时间戳） | ✅ |
| `Assistant Response Preferences` | 基于互动风格推断的回应偏好（带置信度） | ❌ |
| `Notable Past Conversation Topic Highlights` | 高级别主题摘要（带置信度） | ❌ |
| `Helpful User Insights` | 系统推导的事实性描述 | ❌ |
| `Recent Conversation Content` | 近期聊天摘要（~40条，仅用户消息） | ❌ |
| `User Interaction Metadata` | 模型使用比例/账户年龄/设备/意图标签/UI偏好 | ❌ |

### 1.4 关键维度总结

| 维度 | 实现方式 |
|------|----------|
| **记忆存储格式** | JSON 六模块结构化档案（服务端存储） |
| **记忆注入时机** | 每次请求动态挑选 5~20 条最相关信息，合并进 System Prompt |
| **记忆检索方式** | Embedding 语义向量匹配（主要） + 摘要直接注入（辅助） + 全部显式记忆直接注入 |
| **记忆淘汰策略** | 隐式淘汰：语义不匹配自然屏蔽 + 摘要滚动覆盖（~40条/~15条） + 时间精度渐进衰减；无显式遗忘算法披露 |

### 1.5 用户管理界面

- **显式控制**：查看/删除已保存记忆、记忆开关、临时聊天模式、存档聊天
- **隐式控制**：5 个模块完全不可见不可编辑
- 可通过询问"你记得关于我的什么？"主动查询

### 1.6 核心优势

**极致低延迟的记忆人格化**——牺牲详细历史上下文换取速度，用简单优雅的工程方案（Embedding匹配 + 直接注入）替代复杂检索系统。每个用户看到的是"定制版的自己"触发的 ChatGPT。

## 二、Claude 记忆系统

Claude 的记忆系统分两个产品形态：**Claude.ai Projects**（网页版协作工作区）和 **Claude Code**（CLI 编码工具），两者设计哲学不同但互补。

---

### 2A. Claude.ai Projects（网页版）

### 2A.1 架构概览

Projects 为付费用户提供**持久化项目工作区**，通过项目指令 + 知识文件 + 对话历史 + 工具配置四组件协同，实现跨对话的上下文连续性。

### 2A.2 关键维度总结

| 维度 | 实现方式 |
|------|----------|
| **记忆存储格式** | 项目指令（纯文本）+ 知识文件（上传的 PDF/Word/代码/文本）+ 对话历史（云端存储） |
| **记忆注入时机** | 每次在该项目内发起新对话时，项目指令和知识库内容作为上下文注入 |
| **记忆检索方式** | 小数据量：全量直接注入上下文窗口；超出窗口限制时：**自动切换 RAG 检索增强生成模式** |
| **记忆淘汰策略** | 对话历史不自动跨会话共享（需手动引用）；知识文件需手动更新替换；各项目记忆独立隔离 |

### 2A.3 用户管理

- 项目知识库文件手动上传/替换
- 项目指令可随时编辑，即时生效
- 对话可移入/移出项目，改变记忆归属范围
- 支持团队共享项目（Team/Enterprise）

### 2A.4 核心优势

**零代码的项目级 RAG 自动退化**——小数据量时全量注入保持精确，超出限制自动降级为 RAG 检索，用户无需感知模式切换。

---

### 2B. Claude Code（CLI 工具）

### 2B.1 架构概览

采用**六层分层记忆结构**（类似 Git 配置层级），分为两大体系：**CLAUDE.md**（用户手动维护的确定性规则）和 **Auto Memory**（AI 自动积累的经验笔记）。

### 2B.2 六层记忆结构

| 层级 | 存储位置 | 维护者 | 注入策略 | 优先级 |
|------|----------|--------|----------|:---:|
| 组织策略 | `/Library/Application Support/ClaudeCode/CLAUDE.md` | IT/DevOps | 启动全量加载 | 最低 |
| 项目记忆 | `./CLAUDE.md` 或 `./.claude/CLAUDE.md` | 团队（Git共享） | 启动全量加载 | ↑ |
| 项目规则 | `./.claude/rules/*.md` | 团队（Git共享） | **条件加载**（glob匹配） | ↑ |
| 用户记忆 | `~/.claude/CLAUDE.md` | 个人 | 启动全量加载 | ↑ |
| 项目本地 | `./CLAUDE.local.md` | 个人 | 启动全量加载 | ↑ |
| Auto Memory | `~/.claude/projects/<project>/memory/` | Claude 自动 | 启动加载 MEMORY.md 前200行 + 按需加载主题文件 | **最高** |

### 2B.3 存储结构

```
~/.claude/projects/<project>/memory/
├── MEMORY.md          # 索引文件，每次启动加载前 200 行
├── debugging.md       # 调试相关笔记
├── api-conventions.md # API 设计决策
└── ...                # 其他主题分类文件
```

### 2B.4 关键维度总结

| 维度 | 实现方式 |
|------|----------|
| **记忆存储格式** | Markdown 纯文本文件（用户可读可编辑） |
| **记忆注入时机** | 启动全量加载 CLAUDE.md 系列 + Auto Memory 索引前200行；按需加载主题文件；条件加载（glob路径匹配）项目规则 |
| **记忆检索方式** | 全量直接注入（无检索过程） + 按需文件读取 + glob 条件匹配自动附加 |
| **记忆淘汰策略** | 200 行截断触发 Claude 自主精简，内容迁移到独立主题文件；多级开关控制（对话命令/用户设置/项目设置/环境变量）；子智能体记忆物理隔离 |

### 2B.5 核心优势

**从"被动接收指令"到"主动积累经验"**——Auto Memory 让 AI 自动记录项目模式、踩过的坑和用户偏好，形成自我进化的知识库。六层分层设计提供了最高灵活性。

## 三、WorkBuddy

### 3.1 架构概览

WorkBuddy 采用**四层人格/记忆分离模型**，核心设计哲学：稳定度递减、更新频率递增。SOUL.md 是 AI 人格基石极少改动，工作记忆则是高频写入的动态数据。

### 3.2 四层配置模型

| 文件 | 层级 | 核心职责 | 作用域 | 更新频率 |
|------|------|----------|--------|----------|
| **SOUL.md** | 价值观引擎 | 定义 AI 的行事准则和边界 | 全局持久 | 极少 |
| **IDENTITY.md** | 身份标识 | 设定 AI 的名称、物种、风格 | 全局持久 | 很少 |
| **USER.md** | 用户画像 | 记录用户偏好和工作背景 | 全局持久 | 季度更新 |
| **MEMORY.md + 日志** | 工作记忆 | 跨会话上下文和经验积累 | 项目级 | 高频 |

### 3.3 会话启动注入顺序

```
IDENTITY.md → USER.md → SOUL.md → AGENTS.md → global.md → MEMORY.md → YYYY-MM-DD.md
   └────────── 人格层 ──────────┘      └─ 知识层 ─┘  └────── 项目层 ──────┘
```

### 3.4 MEMORY 三层结构

| 层级 | 位置 | 作用 |
|------|------|------|
| **第1层（权威源）** | 用户业务目录 | 手动维护的主档案，所有工作空间从这里恢复记忆 |
| **第2层（任务级）** | `[工作空间]/.workbuddy/memory/MEMORY.md` | 新建任务时从第1层复制，任务结束后同步回去 |
| **第3层（当天日志）** | `[工作空间]/.workbuddy/memory/YYYY-MM-DD.md` | 当天工作记录，定期归档到第1层 |

### 3.5 关键维度总结

| 维度 | 实现方式 |
|------|----------|
| **记忆存储格式** | Markdown 纯文本文件（全局配置 `~/.workbuddy/` + 项目级 `.workbuddy/memory/`） |
| **记忆注入时机** | 每次会话启动时按固定顺序全量注入 System Prompt |
| **记忆检索方式** | 全量直接注入（无检索过程），所有 Markdown 文件内容直接作为上下文 |
| **记忆淘汰策略** | 三层分层管理：权威源手动维护 → 任务级独立隔离 → 日志按日期归档；项目间记忆物理隔离互不污染 |

### 3.6 注意事项

- 记忆恢复**非系统自动**，需在 SOUL.md 中手动配置恢复规则
- 建议 MEMORY.md 权威源放在非系统盘避免重装丢失
- 存在"记忆投毒"风险：提示词注入可能写入异常内容，需定期检查

### 3.7 核心优势

**人格与记忆彻底解耦**——四层分离设计让底层 AI 人格（SOUL.md）和项目记忆（MEMORY.md）完全独立，修改人格影响全局行为，修改记忆仅影响当前项目上下文。

## 四、Cursor / GitHub Copilot（联合分析）

Cursor 和 GitHub Copilot 都属于**IDE 嵌入式 AI 编码助手**，记忆系统有较多共性，放在一起分析。

---

### 4A. Cursor

### 4A.1 架构概览

Cursor 的记忆系统由两部分组成：**代码库语义索引**（向量嵌入 RAG）和 **AI 规则系统**（`.cursor/rules/*.md`）。

### 4A.2 代码库索引

| 维度 | 实现方式 |
|------|----------|
| **索引对象** | 代码库中的每个文件计算一个嵌入向量 |
| **索引触发** | 项目打开时自动初始化，新文件自动增量索引 |
| **检索方式** | 基于向量嵌入的语义搜索（相似度匹配） |
| **存储格式** | 向量嵌入（具体向量数据库未披露），本地存储 |
| **淘汰策略** | 索引可手动删除单个代码库；可选择性索引（`.cursorignore` / `.gitignore` 排除） |

### 4A.3 AI 规则系统

| 维度 | 实现方式 |
|------|----------|
| **规则类型** | 全局规则（Cursor 设置中配置）+ 项目规则（`.cursor/rules/*.md`） |
| **存储格式** | Markdown 文件，支持 glob 路径匹配 + 语义描述 + `@file` 引用链 |
| **注入时机** | 全局规则始终加载；项目规则在匹配文件被引用时自动附加 |
| **检索方式** | glob 模式匹配自动附加；`@file` 引用支持规则链接 |

### 4A.4 核心优势

**代码结构感知的语义索引**——为每个文件计算嵌入向量而非按字符数切分，让 AI 理解代码的语义结构而非文本片段。

---

### 4B. GitHub Copilot

### 4B.1 架构概览

GitHub Copilot 记忆系统由两大组件构成：**Copilot Memory**（结构化记忆存储）和 **工作区语义搜索**（代码库索引 + 多工具搜索链）。

### 4B.2 Copilot Memory

| 维度 | 实现方式 |
|------|----------|
| **记忆类型** | Repository-level facts（仓库级编码约定/架构决策）+ User-level preferences（用户级交互偏好） |
| **存储格式** | 带代码引用（citations）/ 用户引述的结构化记忆（具体存储后端未披露） |
| **注入时机** | 每次交互时自动验证并检索相关记忆 |
| **检索方式** | 引用验证机制：对照当前分支检查引用的代码是否仍存在/有效，仅使用验证通过的记忆 |
| **淘汰策略** | **28 天未使用自动过期**，每次成功使用重置计时器 |

### 4B.3 工作区语义搜索

| 维度 | 实现方式 |
|------|----------|
| **搜索工具链** | 语义搜索（`#codebase`）+ 文本搜索 + Grep + 文件搜索 + 引用追踪 + 目录列表 + 文件读取 |
| **索引来源** | GitHub 仓库索引（云端即时可用）+ Azure DevOps 索引 + 本地自行构建索引 |
| **检索策略** | Agent 自主分析需求后组合多工具迭代搜索（类似开发者手动探索代码库） |
| **隐私隔离** | 仓库 A 的记忆仅作用于仓库 A，不跨仓库泄露 |

### 4B.4 核心优势

**28 天自动过期 + 引用验证双重保障**——记忆既有自动清理机制避免膨胀，又有实时验证确保准确性的设计是商业化产品中最完善的。

## 五、Replit Agent

### 5.1 架构概览

Replit Agent 采用**"在飞行中"的上下文管理**，核心是三层动态架构：**会话轨迹积累** → **LLM 压缩摘要** → **快照引擎**，辅以**多智能体分布式记忆**。最大特点是零配置全自动，但代价是跨会话记忆极为有限。

### 5.2 多层记忆体系

| 记忆类型 | 作用 | 作用域 |
|----------|------|--------|
| **会话轨迹** | 记录动作序列、观察结果、工具调用和决策 | 单次会话 |
| **压缩轨迹** | LLM 生成的精炼摘要，仅保留最相关信息 | 单次会话（压缩后） |
| **快照引擎** | 捕获完整工作区状态（文件/环境/进程） | 单次会话，可恢复/分支 |
| **replit.md** | 持久项目指令（编码风格/命名约定/产品规则） | 跨会话持久 |

### 5.3 多智能体分布式记忆

| 智能体 | 记忆类型 | 职责 |
|--------|----------|------|
| **Manager** | 战略性记忆 | 高层任务规划与协调 |
| **Editor** | 战术性记忆 | 具体代码修改实现 |
| **Verifier** | 评估性记忆 | 测试和验证变更 |

智能体间通过**结构化消息传递**通信，而非共享单一上下文窗口。

### 5.4 关键维度总结

| 维度 | 实现方式 |
|------|----------|
| **记忆存储格式** | 云端会话轨迹（结构化日志）+ LLM 压缩摘要 + 快照（工作区环境镜像）+ replit.md（Markdown） |
| **记忆注入时机** | 会话内持续积累轨迹；replit.md 每会话注入；压缩在任务边界或超出上下文窗口时静默触发 |
| **记忆检索方式** | LLM 自主上下文压缩（LangChain 驱动）——模型在任务边界进行摘要而非死板 Token 阈值触发 |
| **记忆淘汰策略** | 跨会话不保留轨迹/压缩记忆（新会话从头开始）；压缩本身是有损的（可能丢弃后续重要信息）；快照可手动恢复 |

### 5.5 局限性

- **跨会话零记忆**：每次新会话仅从项目文件重新理解代码库
- **压缩有损**：LLM 可能丢弃后续变得重要的上下文
- **用户不可控**：无法查看或影响压缩过程中保留/丢弃的信息
- **无向量存储/数据库**：完全依赖轨迹和快照，缺少语义检索能力

### 5.6 核心优势

**零配置全自动 + 快照引擎**——用户无需任何配置文件，Agent 自动管理上下文。快照引擎提供环境级别的记忆（可回滚、分支探索），是其他产品不具备的能力。

## 六、综合对比总表

| 产品 | 存储格式 | 注入时机 | 检索方式 | 淘汰策略 | 核心优势 |
|------|----------|----------|----------|----------|----------|
| **ChatGPT** | JSON 六模块档案 | 每次请求动态挑选 5-20 条 | Embedding 语义匹配 | 隐式淘汰（语义屏蔽+滚动覆盖） | 极致低延迟记忆人格化 |
| **Claude Projects** | 文本指令+知识文件+对话历史 | 每次项目内对话 | 小数据全量注入/超限自动 RAG | 知识文件手动更新，记忆项目隔离 | 零代码RAG自动退化 |
| **Claude Code** | Markdown（六层分层+Auto Memory） | 启动全量+按需+条件加载 | 全量注入+按需读取 | 200行截断+AI自主精简 | 主动积累经验的自我进化 |

| **Cursor** | 向量嵌入索引+Markdown规则 | 项目规则glob匹配自动附加 | 语义向量搜索 | 手动删除索引，可选性索引 | 代码结构感知的语义索引 |
| **GitHub Copilot** | 结构化记忆+多工具搜索链 | 每次交互验证检索 | 引用验证+语义搜索+文本搜索 | 28天未使用自动过期 | 28天过期+引用验证双重保障 |
| **Replit Agent** | 轨迹+压缩摘要+快照+replit.md | 会话内持续积累 | LLM自主压缩+快照恢复 | 跨会话不保留 | 零配置全自动+快照引擎 |

## 七、设计模式分类

### 7.1 按存储介质

| 类型 | 代表产品 | 特点 |
|------|----------|------|
| **Markdown 文件** | WorkBuddy, Claude Code | 人类可读可编辑，版本控制友好，适合确定性规则 |
| **结构化 JSON/数据库** | ChatGPT, GitHub Copilot | 机器优化，支持复杂查询和验证 |
| **向量嵌入** | Cursor, ChatGPT | 语义检索，适合大规模非结构化内容 |
| **云端轨迹日志** | Replit Agent | 全自动，零用户配置 |

### 7.2 按注入策略

| 类型 | 代表产品 | 特点 |
|------|----------|------|
| **全量注入** | WorkBuddy, Claude Code | 简单可靠，受上下文窗口限制 |
| **语义检索注入** | ChatGPT, Cursor | 高效精选，依赖向量质量 |
| **混合注入** | Claude Projects, Copilot | 灵活适配不同数据规模 |
| **无注入（命令式）** | Replit Agent | Agent 主动调用工具获取 |

### 7.3 按淘汰策略

| 类型 | 代表产品 | 特点 |
|------|----------|------|
| **时间制（TTL）** | GitHub Copilot（28天） | 自动化，简单可靠 |
| **容量制（截断）** | Claude Code（200行） | 控制上下文预算 |
| **语义衰减** | ChatGPT | 渐进式，自然合理 |
| **全量丢弃** | Replit Agent | 每次新会话从零开始 |

## 八、关键发现与设计启示

1. **记忆系统的本质是"上下文预算分配"问题**——所有产品都在有限 Token 窗口内做取舍：全量注入更准确但费 Token，语义检索省 Token 但可能丢失细节。

2. **无单一最优方案**——ChatGPT 牺牲历史细节换速度，Claude Code 主动积累经验要准确性，Replit 全自动零配置牺牲跨会话连续性。选择取决于产品定位。

3. **人格与记忆分离是重要趋势**——WorkBuddy 的四层分离、ChatGPT 的显式/隐式记忆分离、Claude 的 CLAUDE.md vs Auto Memory 分离，都体现了"稳定的人格外壳 + 动态的知识内核"的设计思路。

4. **自动化记忆的可靠性是关键瓶颈**——LLM 压缩有损（Replit）、记忆投毒风险（WorkBuddy）、错误记忆修正困难（ChatGPT），自动化记忆系统需要人工审核和纠错机制。

5. **GitHub Copilot 的 28 天 TTL + 引用验证是目前最完善的自动化记忆方案**——既有自动清理防止膨胀，又有实时验证确保准确，实现了"零维护成本"与"高准确性"的平衡。
